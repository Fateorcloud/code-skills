# 🚀 新项目启动指南（混合方案）

**适用于：** 希望结合 mattpocock/skills + Spec Coding 的开发者

---

## ⚡ 5 分钟快速启动

### Step 1: 创建目录结构

```bash
cd your-project

# 创建基础文件
touch CLAUDE.md CONTEXT.md
mkdir -p docs/adr docs/modules
```

### Step 2: 复制模板文件

从 `STARTUP_TEMPLATE.md` 复制以下内容：
1. ✅ CLAUDE.md 的模板
2. ✅ CONTEXT.md 的模板  
3. ✅ docs/adr/001-initial-setup.md 的模板

### Step 3: 让 AI 帮你初始化

**向 AI 发送（复制粘贴）：**

```
我要开始一个新项目：[你的项目简介]

请帮我初始化混合文档体系：

1. **使用 /grill-me** 主动提问理解我的项目
   
2. **生成 CONTEXT.md**
   - 定义项目核心术语（参考 STARTUP_TEMPLATE.md）
   - 保持精简风格
   
3. **生成 CLAUDE.md**
   - 说明文档体系（CONTEXT.md + ADR + 模块文档）
   - 何时用哪种文档
   - 参考 STARTUP_TEMPLATE.md 的模板
   
4. **生成 docs/adr/001-initial-setup.md**
   - 记录技术栈选择
   - 说明选择原因

开始提问吧！
```

## 🎯 决策树：何时用什么文档？

```
开始开发新功能
  │
  ├─ 是否涉及新术语？
  │    └─ 是 → 先更新 CONTEXT.md
  │
  ├─ 功能是否复杂？
  │    ├─ 简单 → 只用 CONTEXT.md，灵活开发
  │    └─ 复杂 → 创建 docs/modules/XXX.md
  │
  ├─ 是否安全敏感？
  │    └─ 是 → 必须创建详细模块文档
  │
  └─ 是否重大架构决策？
       └─ 是 → 记录 ADR
```

## 🎓 最佳实践

### ✅ DO（推荐做法）

1. **启动阶段用 /grill-me**
   - 让 AI 主动提问，快速建立 CONTEXT.md

2. **保持 CONTEXT.md 精简**
   - 只记录术语定义，不写实现细节
   - 一个术语 = 1-2 句话

3. **复杂模块才写详细文档**
   - 认证、支付、隐私等敏感模块
   - 业务逻辑复杂的核心功能

4. **及时记录 ADR**
   - 技术选型时
   - 重大重构时
   - 解决技术债务时

5. **代码变更后同步文档**
   - 新术语 → 更新 CONTEXT.md
   - 架构变更 → 添加 ADR
   - 接口变更 → 更新模块文档

---

### ❌ DON'T（避免）

1. ❌ **一开始就写所有详细文档**
   - 过早优化，浪费时间
   - 需求可能变化

2. ❌ **CONTEXT.md 写得太详细**
   - 应该精简，详细内容放模块文档
   - 目标：快速查阅术语

3. ❌ **文档和代码不同步**
   - 过时的文档比没有更危险
   - AI 会按错误规范执行

4. ❌ **所有模块都写详细 spec**
   - 简单 CRUD 不需要
   - 增加维护负担

## 🔄 实际案例对比

### 案例：开发用户管理功能

#### ❌ 纯 Spec Coding（过度工程）

```
第 1 天：编写 30 页的 USER.md 文档
第 2 天：AI 按规范实现
第 3 天：需求变更，文档全部重写 😰
```

#### ❌ 无文档（混乱）

```
第 1 天："帮我做个用户管理"
AI：生成代码，命名随意
第 2 天：再次生成，风格不同
第 3 天：代码难以维护 😰
```

#### ✅ 混合方案（平衡）

```
第 1 天：
- /grill-me 理解需求
- 生成 CONTEXT.md：User, Profile, Role
- 快速实现基础 CRUD

第 2 天：
- 发现认证部分复杂
- 创建 docs/modules/AUTH.md
- 严格按规范实现登录

第 3 天：
- 普通功能参考 CONTEXT.md
- 认证功能查阅 AUTH.md
- 灵活 + 严格，完美结合 ✅
```
