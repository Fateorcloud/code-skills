## 🚀 5 分钟快速开始

### 1. 项目初始化

```bash
# 在项目根目录
touch CLAUDE.md
mkdir -p docs/modules docs/archive

# 创建核心文档
touch docs/ARCHITECTURE.md
touch docs/DEVELOPMENT_GUIDE.md
touch docs/PROJECT_STATUS.md
```

### 2. 编写 CLAUDE.md（AI 入口）

```markdown
# 项目名称

## 🚀 快速开始
[一句话介绍项目]

## 📚 核心文档导航

### P0 - 必读（进项目就要读）
| 文档 | 作用 | 更新时间 |
|------|------|----------|
| `docs/ARCHITECTURE.md` | 系统架构 | 2026-06-15 |
| `docs/DEVELOPMENT_GUIDE.md` | 开发规范 | 2026-06-15 |

### P1 - 按需阅读（开发特定功能时）
| 模块 | 文档 | 说明 |
|------|------|------|
| 认证 | `docs/modules/AUTH.md` | 登录/注册/权限 |

## 🤖 AI 协作规范

1. 读取本文件找到相关文档
2. 遵循文档规范执行开发
3. 完成后更新 PROJECT_STATUS.md
```

### 3. 开始使用

向 AI 发送：
```
"请按照 CLAUDE.md 的指引，阅读相关文档后实现 XXX 功能"
```

## ⚡ 高频场景速查

### 场景 1: 开发新功能

```
1. 确定涉及的模块
2. 查看是否有模块文档
   - 有 → 更新文档
   - 无 → 创建 docs/modules/XXX.md
3. 在文档中定义 Spec（功能、接口、流程）
4. 向 AI："请按照 docs/modules/XXX.md 实现 YYY"
5. 审查代码 + 测试
6. 更新 PROJECT_STATUS.md
```

### 场景 2: 修复 Bug

```
1. 在 PROJECT_STATUS.md 记录 Bug
2. 向 AI："请调查 PROJECT_STATUS.md 中的 Bug #X"
3. AI 读取相关文档 + 代码，分析原因
4. AI 提出修复方案，等待确认
5. 修复 + 测试
6. 更新 PROJECT_STATUS.md
```

### 场景 3: 代码审查

```
1. 向 AI："请审查当前分支的代码，检查是否符合规范"
2. AI 读取 DEVELOPMENT_GUIDE.md + 相关模块文档
3. AI 列出不符合规范的地方
4. 逐项修复
```

### 场景 4: 重构代码

```
1. 在 ARCHITECTURE.md 添加 ADR（架构决策记录）
2. 更新相关模块文档
3. 旧文档移到 docs/archive/
4. 向 AI："请按照新的 XXX.md 规范重构模块"
```

## 🚫 常见错误

### ❌ 错误 1: 直接让 AI 写代码，没有 Spec

```
❌ "帮我写个用户登录功能"
✅ "请按照 docs/modules/AUTH.md 实现用户登录功能"
```

### ❌ 错误 2: 写了 Spec 但没告诉 AI

```
❌ [写了 AUTH.md] → "实现登录功能"（AI 不知道要读文档）
✅ [写了 AUTH.md] → "请按照 docs/modules/AUTH.md 实现登录功能"
```

### ❌ 错误 3: 代码改了但文档没更新

```
❌ 修改代码 → 不更新文档 → AI 下次按旧规范执行
✅ 修改代码 → 同步更新文档 → 保持一致
```

### ❌ 错误 4: 文档写得太抽象

```
❌ "实现一个登录接口"（AI 要猜细节）
✅ "POST /api/v1/auth/login，接收 username+password，
    返回 JWT Token，使用 bcrypt 验证密码"
```

## 📐 文档详细度指南

| 类型 | 详细度 | 示例 |
|------|--------|------|
| 架构文档 | 高层概览 | "使用 FastAPI + PostgreSQL + Redis" |
| 开发规范 | 明确规则 | "代码格式化用 Black，行长 88" |
| 模块文档 | 中等细节 | "密码使用 bcrypt，rounds=12" |
| 接口文档 | 详细定义 | "POST /api/v1/xxx，返回 200/400/401" |

**原则：**
- 写到 AI 能独立实现的程度
- 写"做什么"和"约束"，不写"怎么做"（算法细节）

## 🔄 文档冲突解决

当多个文档有冲突时：

```
1. 优先级高的优先（P0 > P1 > P2）
2. 更新时间新的优先
3. CLAUDE.md 中明确标注的权威文档优先
```

在 CLAUDE.md 中明确声明：
```markdown
## 冲突解决规则
当文档有冲突时：
1. 优先级：P0 > P1 > P2
2. 时间：新文档 > 旧文档
3. 有疑问时询问用户
```

## 🎓 黄金法则

### 开发者的 3 个习惯

1. **功能开发前，先写 Spec**
   - 不要直接让 AI 写代码
   - 先在文档中定义清楚"要什么"

2. **代码变更后，立即更新文档**
   - 代码和文档必须同步
   - 过时的文档比没有文档更危险

3. **每次任务完成，更新 PROJECT_STATUS.md**
   - 标记已完成的任务
   - 记录遇到的问题
   - 更新关键指标

### AI 的 3 个原则

1. **先读文档，再写代码**
   - 从 CLAUDE.md 找到相关文档
   - 理解规范后再执行

2. **遇到不确定的，询问用户**
   - 规范冲突时问
   - 缺少规范时问
   - 架构变更时问

3. **完成任务后，更新状态**
   - 更新 PROJECT_STATUS.md
   - 如果规范有变化，更新模块文档
