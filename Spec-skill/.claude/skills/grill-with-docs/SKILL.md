# grill-with-docs — 边拷问边更新文档

## 要做的事

就这个方案的每一个方面反复盘问我，直到达成共识。沿决策树逐条往下，一次解决一个依赖。**每个问题都给出你推荐的答案。一次只问一个问题。** 能查代码得到答案的，就去查代码。

## 领域感知（探索代码时一并做）

### 文件结构

多数仓库是单一上下文：

```
/
├── CONTEXT.md
└── docs/
    └── adr/
        └── 0001-xxx.md
```

若根目录有 `CONTEXT-MAP.md`，说明是多上下文——读它找到每个上下文各自的 `CONTEXT.md`，并推断当前话题属于哪个；不清楚就问。

**懒创建**：没有 `CONTEXT.md` 就在解决第一个术语时再建；没有 `docs/adr/` 就在需要第一个 ADR 时再建。

### 拷问过程中的动作

- **挑战术语表**：用户用的词和 `CONTEXT.md` 里的定义冲突时，当场指出。"你的术语表把『取消』定义为 X，但你现在像在说 Y——到底哪个？"
- **磨锐模糊词**：词含糊或一词多义时，提出一个精确的规范术语。"你说『账户』——是指 Customer 还是 User？这是两个东西。"
- **用具体场景压测**：讨论领域关系时，编造探边界的场景，逼用户把概念之间的边界讲精确。
- **与代码交叉验证**：用户陈述某功能怎么工作时，核对代码是否一致；发现矛盾就掀出来。"你的代码取消的是整个 Order，但你刚说支持部分取消——哪个对？"
- **内联更新 CONTEXT.md**：术语一旦定型，**当场**更新 `CONTEXT.md`，别攒着批量改。格式见 `toolkit/CONTEXT.template.md`（每个术语带 `_Avoid_` 列表）。`CONTEXT.md` **只能是术语表**——绝不放实现细节、不当 spec、不当草稿本。
- **谨慎提议 ADR**：仅当三条**同时**成立才提议：① 难以逆转 ② 没有背景会让人困惑 ③ 是真实权衡的结果。缺一即跳过。要建时走 `/adr`。
