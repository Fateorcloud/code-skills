# spec-check — 文档体检

只读审查，输出文档体系健康报告。**不修改任何文件。**

## 步骤

### 1. 清点
检查核心文件是否存在：`CLAUDE.md`、`CONTEXT.md`（或 `CONTEXT-MAP.md`）、`docs/adr/`。列出存在与缺失。

### 2. 逐项检查

**CLAUDE.md**
- 导航链接指向的文件是否真实存在？（死链 = 问题）
- 是否说明了"何时读哪个文档 / 用哪个技能"？

**CONTEXT.md（术语表纯净度——重点）**
- 有没有混入**实现细节**？（CONTEXT.md 只能是术语表，发现实现细节即标记）
- 有没有**通用编程概念**（超时、错误类型、工具模式）？这些不该收录。
- 有没有定义在写"做什么"而不是"是什么"？
- 代码 / 对话里是否在用某术语 `_Avoid_` 列表里的词？（用词不统一 = 问题）
- 有没有术语已在代码里改名/删除但文档没更新？

**docs/adr/**
- 编号是否连续（四位数 `NNNN-slug.md`）？
- 有没有不满足"难逆 + 反直觉 + 真权衡"却硬记的 ADR（噪音）？
- 有没有互相矛盾、却都标 `accepted` 的决策？

**docs/modules/**
- 文档描述的接口/约束是否和代码一致？（抽查关键模块）

### 3. 交叉验证
重点找**代码 vs 文档的矛盾**——最危险。如文档承诺的安全约束代码没实现、术语表定义与代码行为不符。

### 4. 输出报告

```
## 文档体检报告
🔴 严重（代码与文档矛盾）
- [文件:行] 问题

🟡 过时 / 缺失 / 术语表不纯净
- ...

🟢 建议
- ...
```

按严重度排序，给可操作建议（但不执行；修复交 `/spec-sync` 或用户）。

## 原则
- 只读不改。
- 优先报告"矛盾"类问题，它们最致命。
