# adr — 架构决策记录

## 第一步：判断值不值得记

三条**同时**成立才建 ADR，否则跳过：

1. **难以逆转** —— 以后改主意代价大
2. **没背景会困惑** —— 未来读代码的人会问"为啥这么干？"
3. **真实权衡** —— 有真的备选，你为特定理由选了这个

易逆转 → 别记（反正会改）；不反直觉 → 没人会问为什么；没备选 → 没啥可记。

### 什么算数
- **架构形态**：monorepo、写模型事件溯源 / 读模型投影到 Postgres
- **上下文间的集成方式**：用领域事件而非同步 HTTP
- **带锁定的技术选型**：数据库、消息总线、Auth、部署目标（不是每个库，只挑换起来要花一个季度的）
- **边界 / 归属决策**："Customer 数据归 Customer 上下文，别人只按 ID 引用"
- **故意偏离常规路**："用手写 SQL 不用 ORM，因为 X"——挡住下一个人来"修正"
- **代码里看不到的约束**：合规要求、SLA（响应必须 <200ms）
- **非显然的否决**：考虑过 GraphQL 选了 REST 的微妙理由——否则半年后又有人提 GraphQL

## 第二步：写

**编号**：扫描 `docs/adr/` 取最大号 +1，四位数 `NNNN-slug.md`（如 `0002-postgres-write-model.md`）。`docs/adr/` 不存在就懒创建。

**默认极简**——一到三句话即可：

```md
# {决策的短标题}

{1-3 句：背景是什么、决定了什么、为什么。}
```

价值在于记下"做过这个决策 + 为什么"，**不在于填满模板**。

## 可选段落（仅在有真实价值时加，多数 ADR 用不上）

- **Status** frontmatter（`proposed | accepted | deprecated | superseded by ADR-NNNN`）——决策会被重新审视时
- **Considered Options** —— 被否的备选值得记住时
- **Consequences** —— 有非显然的下游影响时
